## Слои

### 1. Router layer
Роутеры принимают HTTP-запросы, валидируют входные данные, вызывают бизнес-логику через storage-слой и формируют HTTP-ответы.

Роутеры расположены в папке:

```text
app/routers/
```

### 2. Schema layer
Pydantic-схемы описывают:
- формат входных данных;
- формат выходных данных;
- сериализацию ORM-объектов в API-ответ.

Схемы расположены в папке:

```text
app/schemas/
```

### 3. Storage layer
Storage-слой инкапсулирует работу с базой данных. Именно здесь находятся SQLAlchemy-запросы и логика получения/изменения сущностей.

Storage расположен в папке:

```text
app/db/storage/
```

Используется паттерн **Storage + Context**. 
Storage открывает сессию, создаёт `StorageContext`, вызывает переданную функцию, делает `commit()` при успехе, делает `rollback()` при исключении, закрывает сессию.



### 4. Models layer
SQLAlchemy-модели описывают структуру таблиц и связи между ними.

Расположение:

```text
app/db/models/
```

### 5. Core layer
Папка `core` содержит инфраструктурную логику:
- зависимости FastAPI
- security-функции
- reference data
- загрузку reference data

## Авторизация и аутентификация

Cхема входа по одноразовому коду из email.

### Общая логика
1. Пользователь отправляет email.
2. Сервер генерирует шестизначный код.
3. Код сохраняется в таблицу `email_code` с ограниченным сроком жизни.
4. Пользователь отправляет email и код для подтверждения.
5. Если код валиден:
   - код помечается как использованный;
   - пользователь ищется по email;
   - если его нет — создаётся новый;
   - пользователю выдаются JWT access и refresh токены.

### Access token
Используется для доступа к защищённым эндпоинтам.

### Refresh token
Используется для получения нового access token без повторного ввода email-кода.

### Проверка пользователя
В `deps.py` access token:
- извлекается из заголовка `Authorization: Bearer ...`;
- валидируется;
- из токена достаётся `sub`;
- по `sub` из БД загружается пользователь.

### Проверка администратора
После получения текущего пользователя проверяется его `role_id`.

## Работа с базой данных

### Используемая СУБД
PostgreSQL.

### ORM
SQLAlchemy 2.0 style.

### Сессии
Сессия создаётся через `SessionLocal`.

Роутеры не работают с SQLAlchemy-сессией напрямую. Вместо этого используется обёртка `Storage.run(...)`.

Такой подход даёт:
- единое место открытия и закрытия сессии;
- единое место для `commit / rollback`;
- меньше дублирования кода;
- более чистые роутеры.

Сериализацию ORM-объектов выполняется **внутри** `storage.run(...)`, пока сессия ещё активна. Иначе возможен `DetachedInstanceError`.

## Основные роутеры

### `auth.py`
Отвечает за:
- отправку кода;
- подтверждение кода;
- refresh access token.

### `users.py`
Работа с текущим пользователем и пользовательскими данными.

### `places.py`
Публичная работа с местами:
- список мест;
- просмотр одного места.

### `folders.py`
Работа с пользовательскими коллекциями:
- список папок;
- папка с местами;
- создание папки;
- переименование;
- добавление места;
- удаление места из папки.

### `cities.py`
Работа со справочником городов.

### `tags.py`
Работа со справочником тегов.

### `place_statuses.py`
Работа со справочником статусов мест.

### `health.py`
Проверка доступности сервиса.

## Переменные окружения

Проект использует `.env`.

- `DB_PASSWORD` — пароль PostgreSQL
- `DB_NAME` — название базы данных
- `SECRET_KEY` — секрет для подписи JWT
